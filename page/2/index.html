Compiler Chapter3-1

Chapter 3 Context-Free Grammars and Parsing

the parsing process

The parser may be viewed as a function that takes as its input the sequence of tokens produced by the scanner and produces as its output the syntax tree.

The structure of the syntax tree: 取决于语言的syntactic structure

How to treat errors?

  1. Error in the scanner

    Generate an error token and consume the offending character.

  2. Error in the parser

    1. Report an error message
    2. Recover from the error and continue parsing to find a many errors as possible为了一次性找出尽可能多的错误，可以减少Debug时间，有些Error是同一原因导致的
    3. Sometimes, a parser may perform error repair. 弥补缺失的部分，但是不准确，然而可以使得分析继续进行

Context-free grammars

为何叫上下文无关？笛卡尔积左侧只有一个V，相反：context-relative grammar

A Context-Free Gramar is a 4-tuple

  • V is a finite set of nonterminal symbols
  • sigma is a finite set of terminal symbols
  • S is a distinguished nonterminal, the start symbol
  • → is a finite relation, the productions

Derivations and the language defined by a grammar

Let G = (V,sigma,S,→) be a CFG. The “directly derives” relation is defined as { (αAγ, αβγ) | A→β }

Language: the set of strings

L(G) = { s | exp => s }*

The set of all strings of token symbols obtained by derivations from the exp symbol is the language defined by the grammar of expressions.

  • G represents the expression grammar
  • s represents an arbitrary string of token symbols (sometimes called a sentence)
  • The symble *s =>** stands for a derivation consisting of a sequence of replacements as described earlier
  • Grammar rules are sometimes called productions because they “produce” the strings in L(G) via derivations.

Consider 2 rules

  • A -> A a | a 直接左递归 Left Recursive
  • A -> a A | a 直接右递归 Right Recursive

两个表示的同一种Language: {a^n | n is an integer >= 1}, regular expression (a+)，但是文法不一样，如下面的例子

  • A -> A a | b: b, ba, baaa, …
  • A -> a A | b: b, ab, aab, aaab, …

语言的定义受制于文法的设计

Parse Trees and Abstract Syntax Trees

Parse Trees

替换顺序可以最右推导/最左推导

A parse tree corresponding to a derivation is a labeled tree.

用了parse tree以后，推导顺序不再重要了

  • Leftmost/Rightmost derivation(self-study)

parse tree怎么画，注意点：

  1. 数字不要替换，只写number（对于语法分析来说不关心，都只是token，在语义环节才被考虑）
  2. 不用标推导顺序

Abstract Syntax Trees

把具体的数字带入，标记在number下面，加括号()

简化版本：抽象语法树，去掉所有的exp, number等，只保留树的结构以及数字、运算符

  1. Such trees represent abstractions of the actual source code token sequences, and the token sequences cannot be recovered from them (unlike parse trees). 不可逆！
  2. A parse tree is a representation for the structure of ordinary called concrete syntax when comparing it to abstract syntax.
  3. Abstract syntax can be given a formal definition using a BNF-like notation, just like concrete syntax.

树的形态取决于文法定义

2.4 From an NFA to a DFA

the Subset Construction

  1. Compute the e-closure of the start state of M; this becomes the start state.
  2. 穷举所有转移函数
  3. 计算转移后的状态的e-closure
  • double-circle: circles contain finish(accepting) state of M must be double-circle(?)
  • 多种转移函数时，在此课程中可以省略L-state

Minimizing the number of states in DFA

Algorithm:

  1. It begins with the most optimistic assumption possible: it creates two sets
    • One consisting of all the accepting states
    • The other consisting of all the non-accepting states
  2. Given this partition of the states of the states of the original DFA, consider the transitions on each character a of the alphabet.
    • If all accepting states have transitions on a to accepting states
    • If all accepting states have transitions on a to non-accepting states
    • If there are 2 accepting states s and t that have transitions on a that land in different sets
    • If there are 2 accepting states s and t such that s has an a-transition to another accepting state, while t has no a-transition at all (i.e., an error transition)
    • If any further sets are split, we must return and repeat the process form the beginning.

This process continues until

  1. all sets contain only one element (in which case, we have shown the original DFA to be minimal)
  2. until no further splitting of sets occurs.

2.6 Use of Lex to generate a scanner automatically

Input: a text file containing regular expressions, together with the actions to be taken when each expression is matched

Output: contains C source code defining a procedure yylex that is a table-driven implementation of a DFA corresponding to the regular expressions of the input file, and that operates like a getToken procedure.

The first section: definitions

  • any C code that must be inserted external to any function should appear in this section between the delimiters %{ and %}, {Note the order of these characters!}
  • names for regular expressions must also be defined in this section

The second section: rules

These consist of a sequence of regular expressions followed by the C code that is to be executed when the corresponding regular expression is matched.

The third section: auxiliary routines

Routines are called in the second section and not defined elsewhere.

  • yytext就是和number匹配的内容
  • 如果是数字则输出Hex，不是的话原样输出
  • 应用：程序美化器

Compiler Chapter1

Chapter 1 Introduction

Compiler: computer languages

  • Translate one language to another
  • Source Language(input) to target language(output)
  • Source Language: High-level language c or cpp
  • Target Languages: object code, machine code
    First Compiler: FORTRAN language and its compiler, developed by the team at IBM, John Backus
    Programs related to compilers:
  • Interpreters: A language translator. It executes the source program immediately
  • Assemblers: A translator translates assembly language into object mode
  • Linkers: …
  • Loaders, Preprocessors, Editors, Debuggers.
    Source code —-> Scanner —(tokens)–> Parser —(Syntax tree)–> Semantic Analyzer —(Annotated
    tree)–> Source code optimizer —(Intermediate code)–> Code generator —(Target code)–> Target code
    optimizer —–> Target code
    [Literal table // Symbol table // Error handler]
  1. The scanner
    Lexical analysis: input a stream of characters, output tokens.
  2. The parser
    Determine the structure of the program
    Input: the forms of tokens
    Output: a parse tree or a syntax tree
    Parse tree更复杂,更全面 Syntax tree更简洁
    Syntax tree (abstract syntax tree): a condensation of the information contained in the parse tree.
  3. The semantic analyzer
    Static semantics: including declarations and type checking
    Dynamic semantics
    Attribute grammar: computed by the semantic analyzer
  4. The source code optimizer
    Source-level optimization
    Intermediate code: three-address code or P code
    …
  5. The code generator
  6. The target code optimizer: 简化指令流程

Analysis and synthesis: different angles

Front and back end: Front end — the scanner, parser, semantic analyzer, intermediate code synthesis;
Back end — the code generator, some optimization.
Passes

  • Process the entire source program several times
  • A pass consist of several phases
    Language definition and compilers
  • Relation between the language definition and compiler
    Compiler options and interfaces
  • Interfaces with the operating system
  • Provide options to the user for various purposes
    Error handling: static error, execution error
