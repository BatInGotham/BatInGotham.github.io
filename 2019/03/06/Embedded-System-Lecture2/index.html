Embedded System Lecture2

Lecture 2: 系统设计

并行、响应式行为

  • 始终运行：上电后一直运行系统
  • 必须响应连续和混合的事件：事件种类更多（来自于各种传感器、人机交互、网络请求）
  • 实时系统对于响应有底限要求（联系到操作系统中的实时系统）
  • 通常必须并行处理多个独立的任务

嵌入式软件架构模型

  • 桌面、BS有各自独特的架构模型
  • 嵌入式也有自己的架构模型
  • 裸机、RTOS（Real Time OS）、Linux所体现的并不仅是基础平台的不同，更重要的是架构模型的不同
  • Linux的并行性是建立在进程/线程调度基础上
  • 嵌入式系统的并行性是硬件/软件协同实现的

支持并行的MCU硬件

CPU和外设并行工作，其他外设具有单独的时钟，与CPU同时运行。

e.g. DMA：DMA和CPU都可以从内存中搬运数据，CPU的读入效率比DMA更快，但是CPU和DMA并行工作，可以提高系统的吞吐率，但是并不解决延迟问题。

e.g. ADC：模拟数字转换，功能是给测量引脚的电压，给ADC一个读取任务，ADC需要耗时去运作，CPU将任务交给ADC有利于系统的并行工作。

e.g. 串口通讯、定时器、外部事件…

在嵌入式系统中，硬件并行操作需要开发者自己实现，这是因为嵌入式系统与桌面系统的架构模型的差异。

设计约束

硬件：满足功能性能要求的最低硬件——更慢的CPU以省电、更少的存储器以降低成本…

软件：确定的行为和容错能力——失效必须报警的系统、可以降级的系统…

失效处理：系统需要处理可能的失效而不会崩溃

诊断手段：需要能帮助维修人员快速判断问题的手段（嵌入式设备中诊断数据的保存很有限）（判断嵌入式设备是否还“存活”等）

其他约束，比如成本、大小和重量限制、功率和能耗限制、环境

设计约束的影响

比如所用的单片机、编程语言、操作系统（通常没有，如果有则倾向于RTOS）…

做出够好、够快的系统

联系到软件工程

自顶向下 or 自底向上 ? 都不是绝对正确的

  • 体系、构建都需要利用已有的

开发模式

  • 开发风险：新的产品、例外的需求、需求变更
  • 风险选择模式

嵌入式开发过程

  1. 选择：资源的分配
  2. 设计
  3. 调试：区分硬件软件责任（软件或破坏硬件）
  4. 生产：确保系统可以以较高的精度重复制造（设计PCB、自动烧入程序等场合）
  5. 运行和维护

嵌入式开发手段

  • 交叉编译
  • 交叉调试
  • 仿真调试

设计

  • 软件优先考虑，因为软件会提出隐含的硬件需求：CPU？RAM？ROM？
  • GUI？通信？
  • …

软件实现的部分

  • 操作系统
    • 任务调度
    • 资源管理
    • 设备驱动
  • 协议栈
    • TCP/IP
  • 应用软件框架
    • 除基本系统、物理接口、基本逻辑电路，许多由硬件实现的功能都可以由软件实现

双重性部分

  • 算法
  • 按键去抖动
  • 数学运算
  • 无线电设备
